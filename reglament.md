# РЕГЛАМЕНТ ПЛАНИРОВАНИЯ И ОТЧЕТНОСТИ

## 1. ФАЙЛОВАЯ СТРУКТУРА
- /plan.md   (Текущий оперативный план)
- /report.md (Накопительный лог достижений)

## 2. СТАНДАРТ КОММИТОВ
- plan: update tomorrow's agenda
- report: daily sync and analysis
- lab(<discipline>): <description> (для кода в репозитории)
- infra: <description> (конфиги, линукс, скрипты)

## 3. АЛГОРИТМ ВЕЧЕРНЕГО СИНХРОНА
1. Оценка текущего plan.md -> перенос итогов в report.md.
2. Анализ "вылетевших" задач -> назначение их [PRIORITY] на завтра.
3. Совместное (с ИИ) формирование нового plan.md.
4. Синхронный push обоих файлов.

## 4. ПРАВИЛО "1 ЧАС"
Минимальный квант времени на дисциплину — 60 минут. 
Если дисциплина "съела" время другой — это осознанный выбор Архитектора, фиксируемый в отчете.

1. Архитектура «Двух Файлов»: План и Отчет
Вместо свалки из reports/2025-12-24.md, reports/2025-12-25.md и так далее, мы оставляем в корне репозитория два файла-индикатора.

plan.md: Текущий вектор. Здесь всегда только то, что должно быть сделано сегодня. Вечером после планирования ты его обновляешь.

report.md: Хронологический лог (Append-only). Мы не затираем старое, а дописываем сверху. Таким образом, файл превращается в бесконечную ленту твоих достижений, а Git хранит атомарные изменения этой ленты.
2. Регламент коммитов (Commit Convention)
Чтобы твой GitHub выглядел как работа профи, вводим строгий стандарт префиксов. Это поможет тебе быстро фильтровать историю.
plan: — обновление файла plan.md (новые цели на завтра).
report: — фиксация итогов в report.md.
lab: — малые поделки (код в дисциплинах). Примеры: lab(js): array iteration, lab(html): product card.
feat: / fix: — работа над «Большими проектами» (если они в этом репо) или системными утилитами.
infra: — настройки Linux, алиасы, SSH, конфиги Geany/WebStorm.


3. Логика «Приоритетного Сдвига»
Мне нравится твоя безжалостность к причинам. «Устал» — это шум системы. «Приоритет» — это выбор архитектора.
Правило: Если задача из plan.md не перешла в «Done» в report.md, она обязана появиться в plan.md на завтра с пометкой [PRIORITY]. Если она вылетает второй раз — мы проводим аудит: либо задача невыполнима в текущем состоянии, либо ты врешь себе о её важности.
. Структура записи в report.md
Каждая запись должна быть лаконичной:
Timestamp (Когда закончил блок).
Planned vs Reality (Что хотел / Что вышло).
Reason for Shift (Если приоритет сменился).
Reference (Ссылка на коммит с кодом или внешний репо).

4. Архитектура "12 Дисциплин" ЭТОТ ПУНКТ СПОРНЫЙ..! Я БОЛЬШЕ СКЛОНЯЮСЬ РАЗДЕЛИТЬ ПО ДИСЦИПЛИНАМ БУДЕТ МЕНЬШЕ ПАПОК И БОЛЬШЕ ПОРЯДКА...
Мы не будем создавать 12 папок в корне. Мы сгруппируем их по нашим уровням (L0-L4). Это и есть критерий порядка: принадлежность к слою абстракции.
Слой,Дисциплины (примеры),Где хранить в репо
L-1: Substrate,"Архитектура ПК, Логика",/substrate/
L0: Ground,"Linux, Bash, Сети, Git, SSH",/laboratory/l0-ground/

Philosophy,"Мышление, Английский, Психология",/reports/ (в виде заметок)

5. Что делать с проектами? (Критерий разделения)
Здесь важно не допустить архитектурной каши.
Учебные упражнения (Drills): Карточки товара, мелкие функции, тесты конфигов. Всё это живет внутри mastery-journey/laboratory/.
Большие проекты (Projects): Если ты начинаешь строить полноценный планировщик или сайт-портфолио — это отдельный репозиторий.
Зачем: Проект должен иметь свой package.json, свой README и свою историю деплоя.
Связь: В основном репо mastery-journey в отчете за день ты просто ставишь ссылку на коммит в репозитории проекта.
Правило гигиены: Не делай "репозиторий внутри репозитория" (если не используешь git submodules, но тебе это пока не нужно).

6. Золотые правила ведения (Hygiene Rules)
Чтобы твой репозиторий выглядел "солидно" даже при наличии 12 дисциплин, следуй этим правилам:
Правило именования: Только kebab-case (маленькие буквы через дефис). Никаких "My Project" или "Urok_1". Пример: product-card-refactor, tcp-handshake-notes.
Правило README-матрешки: В каждой крупной папке (например, в /frontend) должен быть свой мини-README.md. Там ты кратко пишешь: "Здесь я храню упражнения по верстке". Это создает ощущение продуманной навигации.
Правило ".gitignore": Не пушь лишний мусор. Логи WebStorm (.idea/), временные файлы ОС (.DS_Store) или тяжелые картинки не должны попадать в репо.

4. Как быть с теорией?
Теория без кода — это просто текст.
Если ты прочитал главу по Архитектуре ПК — напиши краткое резюме (5-10 предложений) в папку /substrate/cpu-basics.md.
Если это Английский — новые слова дня идут в ежедневный отчет в /reports/.

### ПРАВИЛА ГИГИЕНЫ РЕПОЗИТОРИЯ

1. КОД: Всегда в `/laboratory/<layer>/<topic-name>`.
2. ТЕОРИЯ: Конспекты в `.md` файлах в соответствующих папках.
3. ОТЧЕТЫ: Каждый день ОБНОВЛЯЕМ файл в plan.md и report.md.
4. КОММИТЫ: Только на английском (feat:, docs:, fix:, chore:).
5. ПРОЕКТЫ: Ссылки на внешние репозитории в отчетах.
